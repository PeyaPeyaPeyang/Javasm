サブルーチンへのジャンプ

{link:jsr} - 0xa8 (168)
{link:jsr_w} - 0xc9 (201)

目的

サブルーチンのエントリポイントへ制御を移し，復帰アドレスをオペランド・スタックへプッシュします。 歴史的には，古いコンパイラが finally ブロックを実装するために使用していました。

{link:jsr} は符号付き 16 ビット分岐オフセットを使用します。 {link:jsr_w} は符号付き 32 ビット分岐オフセットを使用します。 オフセットは現在命令の命令位置を基準とします。

これらの命令は Java SE 6 以降で非推奨（DEPRECATED）であり，現代的なバイトコードでは使用すべきではありません。 後方互換のために残されていますが，新しい検証戦略では拒否される場合があります。 finally 相当の処理には，代わりに構造化例外ハンドラを使用してください。

オペランド:
  • label: 同一メソッド内のバイトコード分岐オフセットへ解決されるターゲットラベル。
  • {link:jsr}: オフセット範囲 -32768..32767。
  • {link:jsr_w}: オフセット範囲 -2147483648..2147483647。
形式:
  • jsr <label>
  • jsr_w <label>
例:
  • jsr cleanup_subroutine（旧来の finally パターン）
  • jsr_w far_cleanup_subroutine（意味は同じでオフセット幅のみ拡張）
スタック効果:
  • Before: ...
  • After: ..., returnAddress（特殊な returnAddress 型，カテゴリ 1）
検証:
  • これらの命令は現代のクラスファイル運用では廃止扱いであり，現在の Java コンパイラは生成しません。
  • 制御フロー整合性と returnAddress 型の利用に関して，厳密な検証整合性が必要です。
  • スタックマップフレーム中心の現代的検証戦略では，旧来サブルーチンパターンが拒否される場合があります。
備考:
  • 現代的なバイトコードで finally 相当の処理を実装する場合は，構造化例外ハンドラを使用します。
  • jsr/jsr_w 自体は定数プール解決を行いません。
  • これらの命令は非推奨であり，新しい検証戦略で拒否される場合があります。
実行時例外とエラー:

これらの命令固有の実行時例外は発生しません。
