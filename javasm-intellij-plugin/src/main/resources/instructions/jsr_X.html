Jump to Subroutine Instructions

{link:jsr} - 0xa8 (168)
{link:jsr_w} - 0xc9 (201)

Purpose

Transfers control to a subroutine entry point and pushes a return address onto the operand stack. Historically, this was used by older compilers to implement finally blocks.

{link:jsr} uses a signed 16-bit branch offset. {link:jsr_w} uses a signed 32-bit branch offset. The offset is relative to the opcode address of the current instruction.

These instructions are DEPRECATED since Java SE 6 and should not be used in modern bytecode. They are retained for backward compatibility and may be rejected by newer verification strategies. Use structured exception handlers for finally-style logic instead.

Operands:
  • label: target label resolved to a bytecode branch offset within the same method.
  • {link:jsr}: offset range -32768..32767.
  • {link:jsr_w}: offset range -2147483648..2147483647.
Format:
  • jsr <label>
  • jsr_w <label>
Examples:
  • jsr cleanup_subroutine (legacy-style finally pattern)
  • jsr_w far_cleanup_subroutine (same semantics with wider offset)
Stack Effect:
  • Before: ...
  • After: ..., returnAddress (special returnAddress type, category-1)
Verification:
  • These instructions are obsolete in modern class-file workflows and are not emitted by current Java compilers.
  • They require strict verifier consistency for control flow and returnAddress type usage.
  • Modern verification strategies centered on stack-map frames MAY REJECT legacy subroutine patterns.
Notes:
  • Use structured exception handlers for finally-style logic in contemporary bytecode.
  • No constant-pool resolution is performed by jsr/jsr_w themselves.
  • These instructions are OBSOLETE and may be rejected by newer verification strategies.
Runtime Exceptions and Errors:

These instructions do not throw any instruction-specific runtime exceptions.
