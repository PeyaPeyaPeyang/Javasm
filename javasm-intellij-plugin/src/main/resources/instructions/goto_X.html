Unconditional Branch Instructions

{link:goto} - 0xa7 (167)
{link:goto_w} - 0xc8 (200)

Purpose

Transfers control unconditionally to a branch target in the same method. These instructions are the basic building blocks for loops, finite-state transitions, and compiler-generated control-flow rewrites.

{link:goto} uses a signed 16-bit branch offset. {link:goto_w} uses a signed 32-bit branch offset. In both cases, the offset is relative to the opcode address of the current instruction.

Operands:
  • label: Assembler-level label resolved to a bytecode branch offset.
  • {link:goto}: offset range -32768..32767.
  • {link:goto_w}: offset range -2147483648..2147483647.
Format:
  • goto <label>
  • goto_w <label>
Examples:
  • goto loop_start (typical backward branch for loops)
  • goto_w far_target (used when the target is out of 16-bit range)
Stack Effect:
  • Before: ..., stack
  • After: ..., stack (no values consumed or produced)
Verification and Control-Flow Rules:
  • The target must be the start of a valid instruction in the same method body.
  • At every control-flow merge, local-variable and operand-stack types must be consistent.
  • Control flow involving uninitialized objects must satisfy constructor-initialization rules.
Notes:
  • Malformed branch structure is typically rejected during verification (for example as VerifyError).
  • Prefer {link:goto} unless distance requires {link:goto_w}.
Verification:
  • Verification ensures that all branch targets are valid instruction offsets within the same method.
  • Verification ensures that control-flow merges have consistent stack and local variable types.
  • Verification ensures that constructor-initialisation rules are satisfied for control flow involving uninitialized objects.
Runtime Exceptions and Errors:

These instructions do not throw any instruction-specific runtime exceptions. Invalid branch targets are rejected during verification.
