Invoke Instance Method

{link:invokevirtual} - 0xb6 (182)

Purpose

Executes an instance (virtual) method on a given object reference. The actual method to be invoked is determined at runtime based on the dynamic type of the object, enabling polymorphism via dynamic dispatch. This instruction is commonly used to invoke overridden methods in a class hierarchy.

Unlike invokespecial (which bypasses dynamic dispatch), invokevirtual consults the method table (vtable) of the runtime class of the object. This allows the most specific implementation to be called, even if the declared type is a superclass or interface.

Operands:
  • methodref (2 bytes): A constant pool index pointing to a CONSTANT_Methodref_info structure, which includes the class and the method's name and descriptor.
Format:
invokevirtual <methodref>
Methodref format:

The symbolic reference is resolved at runtime to identify the target method:

<classname>-><methodname>(<argument_descriptor>)<return_descriptor>
  • <classname>: Fully qualified internal JVM class name (e.g., java/lang/String)
  • <methodname>: Name of the method (e.g., equals, length)
  • <argument_descriptor>: Describes argument types using JVM descriptors (e.g., I, Ljava/lang/Object;)
  • <return_descriptor>: Describes the return type (e.g., V for void, Ljava/lang/String; for String)
Examples:
  • invokevirtual java/lang/String->length()I
  • invokevirtual java/util/List->get(I)Ljava/lang/Object;
  • invokevirtual java/lang/Object->toString()Ljava/lang/String;
Stack Effect:

The operand stack must contain the object reference and method arguments, in left-to-right order.

Before: ..., objectref, arg1, arg2, ..., argN
After: ..., returnValue?

The object reference is consumed, along with all arguments. If the method returns a value, it is pushed back onto the stack. If the return type is void, nothing is pushed. Return category follows descriptor type rules (category-2 for long/double).

Dispatch Insight:
  • Resolution validates the symbolic method reference and access constraints.
  • Dynamic dispatch then selects the most specific applicable implementation in the receiver class hierarchy.
  • This two-step model is central to understanding polymorphic behaviour and late failures.
Tutorial Pattern:
  • A common safe sequence is instanceof, then branch, then checkcast, then invokevirtual.
  • When transforming bytecode, preserve exact receiver position and descriptor slot accounting.
Notes:
  • The resolved target must be an instance method suitable for virtual invocation.
  • <init> and <clinit> are not invoked via invokevirtual.
  • Use invokespecial for constructors, private methods, and explicit super calls.
Verification:
  • Verification ensures the operand stack has a reference-compatible object and arguments matching the method descriptor.
  • Linkage errors occur if the symbolic reference cannot be resolved or if the resolved method is not accessible.
Runtime Exceptions and Errors:
  • NullPointerException: if the object reference is null.
  • IncompatibleClassChangeError: if the resolved method is static or if the resolved class is an interface.
  • AbstractMethodError: if the resolved method is abstract and the actual class of the receiver does not provide an implementation.
  • LinkageError: if the symbolic method reference cannot be resolved or if the resolved method is not accessible.
  • WrongMethodTypeException: if the actual argument types do not match the method descriptor at runtime.
