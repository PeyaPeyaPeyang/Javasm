Load Constant Instructions

{link:ldc} - 0x12 (18)
{link:ldc_w} - 0x13 (19)
{link:ldc2_w} - 0x14 (20)

Purpose

Loads a constant from the run-time constant pool onto the operand stack. The permitted constant kind depends on the opcode and the constant's category.

Operands:
  • ConstantIndex: constant-pool index. The constant type determines which opcode is valid:
    • {link:ldc}: 8-bit index; loads category-1 constants (for example int, float, String, class literal, method type, method handle, or category-1 dynamic constant).
    • {link:ldc_w}: same constant kinds as {link:ldc}, but with 16-bit index.
    • {link:ldc2_w}: 16-bit index; loads category-2 constants (long, double, or category-2 dynamic constant).
Format:
  • ldc <ConstantIndex>
  • ldc_w <ConstantIndex>
  • ldc2_w <ConstantIndex>
Examples:
  • ldc 10 (loads a category-1 constant from CP index 10)
  • ldc_w 300 (same semantics, wider index encoding)
  • ldc2_w 5 (loads a category-2 constant, such as long or double)
Stack Effect:
  • ldc, ldc_w
    Before: ...
    After: ..., value (one category-1 value)
  • ldc2_w
    Before: ...
    After: ..., value (one category-2 value occupying two stack slots)
Verification:
  • Using an opcode with an incompatible constant kind is rejected by verification/linking.
  • Class, method type, method handle, and dynamic constants may trigger resolution/linkage activity.
  • Dynamic constants are computed via bootstrap methods; linkage failures are reported as linkage errors.
  • No operand-stack inputs are consumed by these instructions.
Notes:
  • {link:ldc} vs {link:ldc_w} differs only by index width, not by loaded value semantics.
  • Prefer mnemonic examples that show constant category, because category drives verifier behaviour.
